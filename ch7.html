Pixels & Resolution
🖼️
What is a Digital Image?
الشرح بالعربي الصورة الرقمية تتكون من عدد كبير من عناصر الصورة تُسمى بكسل (Pixels). كلما زاد عدد البكسلات كانت الصورة ذات دقة أعلى وجودة أفضل. طريقة تخزين الصور تُسمى Bitmap وهي نفس طريقة تخزين محتويات الشاشة في ذاكرة الفيديو.
  • A digital image consists of many picture elements (pixels).
  • The number of pixels determines image quality (resolution) — more pixels = better quality.
  • A bit-map stores graphic data the same way a monitor stores data in video memory.
📐
Resolution & Pixel Depth
الشرح بالعربي دقة الصورة = العرض × الارتفاع (بالبكسلات). عمق البكسل = عدد البتات لكل بكسل ويحدد عدد الألوان = 2 أس عدد البتات. مثال: عمق 8 بت = 256 لون.
📏 Resolution
  • Width × Height (in pixels)
  • e.g. 1920 × 1080 = Full HD
🎨 Pixel Depth
  • 8-bit, 16-bit, 24-bit, or n-bit
  • Colors = 2n
📝 Exam Example A picture of 100×100 pixels with pixel depth of 8 bits. What is the size?
100 × 100 × 8 = 80,000 bits = 80 Kbits = 10 KB (since 8 bits → 1 Byte)

Image Types: Binary, Grayscale & Color
⬛
Binary Image
الشرح بالعربي صورة ثنائية — كل بكسل إما أسود (0) أو أبيض (1). تستخدم بت واحد فقط لكل بكسل.
  • Only 2 values: 0 = black, 1 = white
  • 1-bit pixel depth
  • Used in logos, text, documents
🔲
Grayscale Image
الشرح بالعربي صورة رمادية — 8 بت = 256 درجة من الرمادي. كل بكسل له شدة ضوء واحدة.
  • Single intensity value per pixel
  • Range: 0 (black) → 255 (white)
  • Common: 8 bits = 256 shades
  • 2-bit: 00=black, 01=dark gray, 10=light gray, 11=white
🌈
RGB Color Image
الشرح بالعربي كل بكسل في صورة RGB يتكون من ثلاث قيم: Red + Green + Blue. دمج القيم الثلاثة ينتج الألوان. اللون الحقيقي 24-bit = 8 bits لكل قناة = 16.7 مليون لون.
  • Each pixel has 3 channels: R, G, B
  • 24-bit: 8 bits per channel → 16.7 million colors
  • Additive color model — adding light
  • Device-dependent: same RGB looks different on different screens
04
Image Formats: Bitmap vs Vector
🗺️ Bitmap (Raster)
  • Stores pixels directly
  • Disadvantage: loses quality when enlarged
  • Examples: BMP, JPEG, PNG, GIF
📐 Vector
  • Stored as equations: start point, length, direction
  • Advantage: scales without quality loss, smaller size
  • Examples: SVG, AI (similar to MIDI for audio)
📄
GIF Format
الشرح بالعربي صيغة GIF مناسبة للصور ذات الحواف الحادة كالشعارات. تدعم الرسوم المتحركة. محدودة بـ 256 لون — لا تُستخدم للصور الفوتوغرافية.
  • Suitable for sharp-edged line art (logos) with limited colors
  • Uses lossless compression
  • Can store small animations and low-res film clips
  • Max 256 colors → NOT suitable for photography
  • GIF, JPEG, PNG can be viewed directly in a web browser

RGB Color Model
الشرح بالعربي نموذج RGB تضافري: الأحمر + الأخضر + الأزرق تُضاف معاً لإنتاج ألوان مختلفة. يعتمد على الجهاز — نفس قيمة RGB تظهر بألوان مختلفة على أجهزة مختلفة. في بث التلفزيون: الإشارة تُقسَّم لـ Luminance (سطوع) + Chrominance (لون) للتوافق مع الأجهزة القديمة.
  • Red, Green, Blue light added together = additive model
  • Device-dependent — same RGB value looks different on different screens
  • TV broadcast: signal split into Luminance + Chrominance
  • Input devices: TV cameras, scanners, digital cameras
  • Output devices: CRT, LCD, plasma, monitors, JumboTron

Data Compression — Overview
📦
Why Compression?
الشرح بالعربي البيانات غير المضغوطة تحتاج مساحة هائلة. تلفزيون PAL يحتاج 216 ميجابت/ثانية! لذا الضغط ضرورة لا رفاهية.
📝 Video Size Example 100×100 RGB video at 24 fps for 1 minute:
100 × 100 × 24 bits × 24 frames × 60 seconds ÷ 8 = 43.2 MB
✅
Lossless Compression
الشرح بالعربيضغط بدون فقدان — يمكن استعادة البيانات الأصلية بالكامل. مطلوب للصور الطبية.
  • Decompressed data exactly matches original
  • Entropy encoding is lossless
  • Required for: X-Rays, legal docs
  • Exploits statistical redundancy
⚠️
Lossy Compression
الشرح بالعربيضغط مع فقدان — بعض البيانات غير الضرورية تُفقد ولا يمكن استعادتها. يعطي نسب ضغط أعلى.
  • Some insignificant data lost when decompressing
  • Source encoding is usually lossy
  • JPEG, MPEG — higher compression ratio
  • Not reversible

Run-Length Encoding (RLE)
🔁
How RLE Works
الشرح بالعربي ترميز طول التشغيل: تسلسلات البايتات المتكررة تُستبدل بعدد تكراراتها وقيمتها. بدلاً من AAAA نخزن (A,4). يُستخدم كثيراً في ضغط الصور وهو جزء من pipeline ضغط JPEG. في أسوأ حالة (ضوضاء عشوائية) قد ينتج ملفاً أكبر!
  • Content-dependent coding — replaces repeated bytes with (count, value)
  • A special flag byte is used (one not in the data stream)
  • If a byte occurs ≥ 4 consecutive times: count = occurrences − 4
  • Can represent 4 to 259 bytes with just 3 bytes
RLE Example Original: BBBBWWWYYYYYYGGGGYYY
Encoded: (B,4) , (W,3) , (Y,6) , (G,4) , (Y,4)
⚠️ Worst Case For random noise images — RLE can produce a larger file than the original!

🌳 Huffman Encoding — Detailed
💡
What is Huffman Encoding?
الشرح بالعربي ترميز هوفمان هو خوارزمية ضغط بدون فقدان (Lossless) تعتمد على إحصائيات تكرار الرموز. الفكرة الأساسية: الحروف الأكثر تكراراً تأخذ رموزاً أقصر، والحروف الأقل تكراراً تأخذ رموزاً أطول. الناتج كود خالٍ من البادئات (Prefix-Free) مما يضمن فك الضغط الصحيح دون غموض. يُبنى شجرة ثنائية تُحدَّد منها الرموز.
  • Lossless compression — original data fully recoverable
  • Statistical encoding: both sender and receiver know occurrence probabilities
  • Frequent characters → shorter codes | Rare characters → longer codes
  • Successive characters coded independently
  • Result is Prefix-Free → unique, unambiguous decoding guaranteed
  • A binary tree is built: leaves = characters, internal nodes = combined probabilities, edges = 0 or 1
🔑
Why Prefix-Free Codes Matter
الشرح بالعربي في كود Huffman، لا يكون رمز أي حرف هو بداية (prefix) رمز حرف آخر. هذا يضمن فك الضغط الصحيح — عند القراءة من اليسار لليمين يمكن تحديد كل حرف بوضوح بدون غموض. مثلاً: إذا B=1 وA=010، فالتسلسل 1010 = B,A وليس غيره.
Why prefix-free matters — example Codes: B=1, A=010, D=011, E=001, C=000

Decoding "1010010":
1 → B
010 → A
010 → A
Result: BAA ✓ (no ambiguity possible)
📋
Huffman Algorithm — Step by Step
خوارزمية هوفمان — خطوة بخطوة 1. رتّب الحروف تصاعدياً حسب احتمالياتها (الأقل أولاً). 2. ادمج أقل اثنين في عقدة جديدة. 3. كرّر حتى تبقى عقدة جذر واحدة (احتمالها = 1.0). 4. اعطِ 0 للفرع الأيسر و1 للأيمن (أو العكس باتساق). 5. اقرأ الكود من الجذر إلى كل ورقة.
1
Sort by Probability (Ascending)
List all characters ordered from lowest to highest probability. The two lowest will be combined first.
2
Combine Two Lowest
Take the two nodes with the smallest probabilities. Combine them into a new parent node whose probability = sum of the two children.
3
Re-insert and Repeat
Insert the new combined node back into the sorted list. Repeat steps 1–2 until only one node remains (the root, with probability = 1.0).
4
Assign Edge Labels (0 and 1)
Label every left edge as 0 and every right edge as 1 (or vice versa, consistently). This is done on the final tree.
5
Read Code for Each Character
For each character (leaf node), read the bits along the path from the ROOT down to that leaf. That bit-string is its Huffman code.
🧮
Worked Example — Full Solution
مثال محلول بالكامل الحروف A,B,C,D,E مع احتمالياتها: B=0.51 (الأعلى) يأخذ أقصر رمز "1". C=0.09 (الأقل) يأخذ أطول رمز "000". متوسط طول الرمز = 1.98 بت/رمز مقارنةً بـ 8 بت في ASCII.
Given: Characters A, B, C, D, E A: p = 0.16  |  B: p = 0.51  |  C: p = 0.09  |  D: p = 0.13  |  E: p = 0.11
Step 1 — Sort ascending: C(0.09), E(0.11), D(0.13), A(0.16), B(0.51) Step 2 — Combine C+E = CE (0.20). New list: D(0.13), A(0.16), CE(0.20), B(0.51)
Step 3 — Sort: D(0.13), A(0.16), CE(0.20), B(0.51). Combine D+A = DA(0.29)
Step 4 — Sort: CE(0.20), DA(0.29), B(0.51). Combine CE+DA = CEDA(0.49)
Step 5 — Combine CEDA+B = Root(1.00) ✓
🌳 Huffman Tree Visualization
Final Huffman Codes B = 1       (1 bit — most frequent, p=0.51)
A = 011     (3 bits — p=0.16)
D = 010     (3 bits — p=0.13)
E = 001     (3 bits — p=0.11)
C = 000     (3 bits — least frequent, p=0.09)
Average Code Length Calculation L̄ = Σ (bits × probability)

= (1×0.51) + (3×0.16) + (3×0.13) + (3×0.11) + (3×0.09)
= 0.51 + 0.48 + 0.39 + 0.33 + 0.27
= 1.98 bits/symbol
📊 Compression Comparison — 1000 Characters Huffman: 1000 × 1.98 = 1,980 bits ≈ 248 Bytes
ASCII: 1000 × 8 bits = 8,000 bits = 1,000 Bytes (1 KB)
Saving: ~75% size reduction!

Encoding & Decoding Process
الشرح بالعربي الترميز: استبدل كل حرف في النص بكوده الثنائي. فك الترميز: ابدأ من جذر الشجرة، اقرأ بتاً بتاً: 0 = اذهب يساراً، 1 = اذهب يميناً. عند الوصول لورقة (حرف) اكتبه وعُد للجذر.
🔒 Encoding
  • Replace each character with its Huffman code
  • "BAC" → 1 011 000
  • Frequent chars → fewer bits total
🔓 Decoding
  • Start at tree root
  • 0 = go left, 1 = go right
  • On reaching a leaf → write character, return to root
Decoding Example — Input: 1 011 001 1 → go right → reach leaf B → output: B, reset
0 → go left
1 → go right
1 → go right → reach leaf A → output: A, reset
0 → go left
0 → go left
1 → go right → reach leaf E → output: E
Result: BAE
📌 Huffman Key Properties — Summary
✅ Advantages
• Optimal prefix-free code
• Lossless — no data lost
• Simple to implement
• Variable-length codes adapt to data frequency
⚠️ Limitations
• Both encoder and decoder need the same probability table
• If probabilities change, the tree must be rebuilt
• Overhead: must transmit the tree/table along with data
• Less efficient than arithmetic coding for very skewed distributions

🖼️ General — الأسئلة العامة
Q01
A picture is 200×150 pixels with a pixel depth of 16 bits. What is the size of this picture in KB?
200 × 150 × 16 = 480,000 bits ÷ 8 = 60,000 bytes = 60 KB
Q02
What is the difference between a bitmap image and a vector image? Give one advantage and one disadvantage of each.
Bitmap: stores pixels — disadvantage: loses quality when enlarged. Advantage: easy to display.
Vector: stores equations — advantage: scales without quality loss, smaller size. Disadvantage: not suitable for complex photos.
Q03
What is the difference between Lossless and Lossy compression? Give an example use case for each.
Lossless: decompressed data exactly matches original. Used for: medical X-Rays, legal documents. Examples: RLE, Huffman.
Lossy: some data is permanently lost. Higher compression ratio. Used for: JPEG photos, MP3 audio, MPEG video.
Q04
An uncompressed RGB video is 100×100 pixels at 24 fps. How much storage does 1 minute require?
100 × 100 × 24 bits × 24 fps × 60 seconds ÷ 8 = 43,200,000 bytes ≈ 43.2 MB
Q05
Why is GIF not suitable for photographic images?
GIF supports a maximum of 256 colors. Photographs require thousands or millions of colors for accurate representation. Digital photographers use JPEG, TIFF, or RAW instead.
Q06
Why is the RGB color model described as "device-dependent"?
The same RGB value looks different on different devices because each device uses different phosphors, dyes, or display technology. A color that appears bright red on one monitor may appear slightly orange on another.
Q07
Encode the following row using RLE: AAAABBBBBBCCCDDDD
(A,4), (B,6), (C,3), (D,4)
Original: 17 bytes. Encoded: 4 pairs × 2 = 8 bytes. Good compression here!
Q08
Describe the digital camera image capture pipeline in order.
1. Shutter opens briefly to expose the lens to light.
2. Light falls on the CCD array where intensity is stored as voltage.
3. ADC (Analog-to-Digital Converter) converts the charge to a digital value.
4. Digital image stored in memory for viewing or transfer via I/O interface.
🌳 Huffman Encoding — أسئلة هوفمان
H01
What is the core idea behind Huffman encoding? How does it achieve compression?
Huffman assigns shorter binary codes to more frequent characters and longer codes to rare characters. Since frequent characters use fewer bits, the total number of bits in the encoded message is reduced. The result is a lossless, prefix-free code.
H02
Given: A(p=0.16), B(p=0.51), C(p=0.09), D(p=0.13), E(p=0.11). Build the Huffman tree and find all codes.
Codes: B=1, A=011, D=010, E=001, C=000
Average length: (1×0.51)+(3×0.16)+(3×0.13)+(3×0.11)+(3×0.09) = 0.51+0.48+0.39+0.33+0.27 = 1.98 bits/symbol
H03
Using the codes from H02, encode the message "BADE".
B=1, A=011, D=010, E=001
Encoded: 1 · 011 · 010 · 001 = 1011010001 (10 bits)
H04
Using the codes from H02, decode the bit string: 1 001 000 011
1 → B | 001 → E | 000 → C | 011 → A
Decoded: BECA
H05
What does "Prefix-Free" mean in Huffman codes? Why is this property important?
Prefix-free means no code is the beginning (prefix) of another code. Example: if B=1, no other code starts with 1 alone. This guarantees that when decoding a bit stream from left to right, each character is uniquely identifiable — there is no ambiguity about where one character ends and the next begins.
H06
If a text has 500 characters with the probabilities from H02, how many bytes does Huffman encoding use vs ASCII?
Huffman: 500 × 1.98 = 990 bits ≈ 124 Bytes
ASCII: 500 × 8 = 4,000 bits = 500 Bytes
Saving: 376 Bytes (75% reduction)
H07
Given: X(p=0.5), Y(p=0.3), Z(p=0.2). Build the Huffman tree and calculate the average code length.
Sort: Z(0.2), Y(0.3), X(0.5). Combine Z+Y = ZY(0.5). Combine ZY+X = Root(1.0)
Codes: X=0, Y=11, Z=10
Avg Length: (1×0.5)+(2×0.3)+(2×0.2) = 0.5+0.6+0.4 = 1.5 bits/symbol
H08
What information must both sender and receiver share for Huffman to work?
Both must know the probability/frequency table of the characters (or equivalently, the Huffman tree itself). If the receiver doesn't have the tree, they cannot decode the bit stream. This is a key overhead — the table/tree must be transmitted alongside the compressed data.
H09
In the Huffman algorithm, when building the tree, which two nodes are combined at each step?
At each step, the two nodes with the lowest probabilities are combined into a new parent node. The parent node's probability equals the sum of the two children's probabilities. This is repeated until only one node (the root) remains.
H10
Compare Huffman encoding vs RLE. When would you prefer each?
RLE: best when data has long runs of repeated values (e.g., simple images, logos, horizontal areas of same color). Fails with random/noisy data — can make files larger.
Huffman: best when character frequencies are skewed (some much more common than others). Works on any data type. Always produces optimal prefix-free codes. Often used inside JPEG as part of its compression pipeline.
